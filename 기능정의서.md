# KORION KORI Backend 기능정의서

## 1. 목적
- `개발.md` 기준 지갑/원장/운영통제 기능을 현재 코드 기준으로 정리한다.
- 운영자와 개발자가 어떤 기능이 실제 구현되어 있는지 빠르게 파악할 수 있도록 한다.
- API, 원장 상태, 운영 도구를 기능과 상세기능 단위로 구분한다.

## 2. 범위
- 백엔드 API 서버
- 내부 원장(Ledger)
- 입금/출금/내부이체
- 운영 통제(리스크, 승인, 감사로그, 큐)
- 시스템 모니터링/대사/sweep 기록

## 3. 기능 목록
| 기능 | 상세기능 |
| --- | --- |
| Wallet Core | 주소 바인딩, 잔액 조회, 내부 이체 |
| Deposit Core | 수동 입금 반영, 추적 주소 필터, 중복 방지, foxya 연동 자동 입금 모니터 |
| Withdraw Core | 출금 요청, 승인, 브로드캐스트, 확정, 실패 복구 |
| Risk Control | 1회/1일 한도, 요청 리스크 평가, 승인 수 계산 |
| Admin Control | 승인 기록, 대기 승인 조회, 감사 로그 |
| Scheduler | 출금 큐 처리, 수동 검토 전환, stuck 재처리, receipt reconcile |
| Monitoring | system wallet 잔액 수집, 수집 이력 저장, 상태 조회 |
| Reconciliation | 내부 부채 합계 vs 추적 지갑 잔액 비교, 핫월렛 임계치 알림 |
| Sweep Ops | sweep 계획 생성, sweep 기록 조회, 수동 브로드캐스트/확정 기록 |
| Sandbox/On-chain Tools | 네트워크별 잔액 조회, 핫월렛 직접 전송 테스트 |

## 4. 상세 기능

### 4.1 Wallet Core
- 목적
	- 사용자 계정과 지갑 주소를 연결하고 내부 원장 잔액을 조회/이동한다.
- 상세기능
	- `POST /api/wallets/address-binding`
		- `userId`와 `walletAddress`를 1:1로 바인딩한다.
	- `GET /api/wallets/address-binding`
		- `userId` 또는 `walletAddress` 기준으로 바인딩 정보를 조회한다.
	- `GET /api/wallets/balance`
		- 가용 잔액과 locked 잔액을 반환한다.
	- `POST /api/wallets/transfer`
		- 내부 원장 간 이동을 처리한다.
		- `Idempotency-Key`로 중복 호출을 제어한다.

### 4.2 Deposit Core
- 목적
	- 블록체인에서 감지된 입금을 내부 원장에 반영한다.
- 상세기능
	- `POST /api/deposits/scan`
		- `txHash`, `toAddress`, `amount`, `blockNumber`를 받아 입금을 반영한다.
		- `userId` 또는 `walletAddress` 기준으로 계정을 식별한다.
	- 추적 주소 필터
		- 재단/입금/핫 지갑으로 설정된 주소만 입금 반영 대상이다.
	- 중복 방지
		- 동일 `txHash`는 1회만 ledger 반영된다.
	- 자동 입금 모니터
		- foxya 내부 API `GET /watch-addresses`로 감시 주소를 조회한다.
		- TRON KORI `Transfer` 이벤트를 주기 스캔한다.
		- 감지 시 foxya 내부 API `POST /register`를 호출한다.
		- 요구 컨펌 수 도달 시 foxya 내부 API `POST /:depositId/complete`를 호출한다.
		- 감지 cursor와 event fingerprint를 DB에 저장해 재시작 후에도 중복 등록하지 않는다.

### 4.3 Withdraw Core
- 목적
	- 내부 부채 차감과 온체인 전송 상태를 분리해 출금 수명주기를 관리한다.
- 최종 책임
	- 상태머신, 원장 반영, BullMQ 재시도, recover seed는 `coin_manage`가 소유한다.
	- 관리자 승인/거절 UI는 `coin_csms`가 소유한다.
	- 레거시 `foxya_coin_service`, `coin_publish`는 호환 API 또는 참고 구현으로만 둔다.
- 상세기능
	- `POST /api/withdrawals`
		- 출금 요청을 생성한다.
		- 요청 시 가용 잔액이 차감되고 locked 잔액으로 이동한다.
	- `POST /api/withdrawals/{withdrawalId}/approve`
		- 승인 기록을 남기고 요구 승인 수를 충족하면 `approved`로 전이한다.
	- `POST /api/withdrawals/{withdrawalId}/broadcast`
		- 핫월렛 전송 이후 `txHash`를 기록한다.
	- `POST /api/withdrawals/{withdrawalId}/confirm`
		- 온체인 확정 시 locked 잔액을 소거하고 ledger tx를 `confirmed`로 전환한다.
	- 실패 복구
		- 출금 실패 시 locked 잔액을 다시 가용 잔액으로 복구한다.
	- queue recovery
		- 앱 재기동 시 `approved`, `broadcasted` 상태 건을 다시 queue seed 한다.
	- failed job 운영
		- 실패한 dispatch/reconcile job을 조회하고 recover 할 수 있다.

### 4.4 Risk Control
- 목적
	- 출금 요청 시 운영 리스크를 계산하고 승인 강도를 결정한다.
- 상세기능
	- 한도 검사
		- `WITHDRAW_SINGLE_LIMIT_KORI`
		- `WITHDRAW_DAILY_LIMIT_KORI`
	- 요청 리스크 평가
		- 금액 크기
		- `clientIp` 존재 여부
		- `deviceId` 존재 여부
		- 외부 IP 여부
	- 결과 필드
		- `riskLevel`
		- `riskScore`
		- `riskFlags`
		- `requiredApprovals`

### 4.5 Admin Control
- 목적
	- 운영 승인과 추적 로그를 남겨 출금 통제를 가능하게 한다.
- 상세기능
	- `GET /api/withdrawals/pending-approvals`
		- 아직 요구 승인 수를 채우지 못한 출금을 조회한다.
	- `GET /api/withdrawals/{withdrawalId}/approvals`
		- 승인자, 승인 시각, 메모를 조회한다.
	- `GET /api/system/audit-logs`
		- `withdrawal`, `sweep`, `system` 엔티티 기준 감사 로그를 조회한다.
	- 감사 로그 이벤트
		- 출금 요청
		- 승인 부분완료/최종완료
		- 브로드캐스트
		- 확정
		- 실패
		- 리뷰 필요 전환
		- sweep 계획/브로드캐스트/확정

### 4.6 Scheduler
- 목적
	- 자동 승인 가능한 건은 처리하고, 사람이 봐야 하는 건은 review queue로 보낸다.
- 상세기능
	- `POST /api/scheduler/process-withdraw-queue`
		- 저위험 1단계 승인 건은 시스템 계정으로 자동 승인한다.
		- 중위험/고위험 건은 `review_required`로 전환하고 수동 검토 job을 생성한다.
	- `POST /api/scheduler/retry-pending`
		- 오래된 출금 상태를 재점검한다.
		- `broadcasted` 상태는 receipt 기준으로 reconcile 한다.
		- 나머지 상태는 수동 검토 queue로 넘긴다.

### 4.7 Monitoring
- 목적
	- 운영 지갑 잔액과 수집 상태를 저장하고 상태 API에서 노출한다.
- 상세기능
	- 백그라운드 수집기
		- system wallet들을 주기적으로 읽어 최신 snapshot을 저장한다.
	- 수동 수집
		- `POST /api/system/monitoring/run`
	- 상태 조회
		- `GET /api/system/status`
		- 수집 결과, runtime, wallet catalog, 대사 요약을 함께 제공한다.
	- 입금 모니터 상태 조회
		- `GET /api/system/deposit-monitor`
		- 활성화 여부, currency filter, 최근 이벤트, cursor를 제공한다.
	- 입금 모니터 수동 실행
		- `POST /api/system/deposit-monitor/run`

### 4.8 Reconciliation
- 목적
	- 내부 원장 부채와 추적 지갑 잔액을 비교해 운영 상태를 빠르게 판단한다.
- 상세기능
	- `GET /api/system/reconciliation`
		- 내부 잔액 합계
		- locked 잔액
		- 총 부채
		- 추적 지갑 총 KORI 잔액
		- 차이값(`covered`/`deficit`)
		- 핫월렛 임계치 알림

### 4.9 Sweep Ops
- 목적
	- sweep 필요 잔액을 기록하고 foxya completed deposit 기준 자동 sweep을 수행한다.
- 상세기능
	- `POST /api/system/sweeps/plan`
		- 최신 모니터링 snapshot 기준으로 `treasury`, `hot`를 제외한 지갑에서 sweep 계획을 생성한다.
		- 이미 `planned` 또는 `broadcasted` 상태인 동일 source wallet은 중복 계획하지 않는다.
	- `GET /api/system/sweeps`
		- sweep 기록 조회
	- `POST /api/system/sweeps/{sweepId}/broadcast`
		- 수동 브로드캐스트 txHash 기록
	- `POST /api/system/sweeps/{sweepId}/confirm`
		- 수동 확정 기록
	- `GET /api/system/sweep-bot`
		- foxya deposit 기반 자동 sweep bot 상태 조회
	- `POST /api/system/sweep-bot/run`
		- foxya completed deposit를 조회해 source wallet private key로 hot wallet 집금을 시도한다.
		- 성공 시 foxya internal API `/:depositId/sweep/submit` 호출
		- 실패 시 foxya internal API `/:depositId/sweep/fail` 호출
		- local sweep record와 audit log를 같이 남긴다.
	- `POST /api/system/alerts/telegram/test`
		- 텔레그램 알림 채널 연동 상태 확인용 테스트 메시지를 발송한다.
		- request body에 `message`를 넣으면 사용자가 입력한 문구로 발송한다.
	- `GET /api/system/external-alert-monitor`
		- 외부 health target/foxya DB 이벤트 모니터의 현재 상태, cursor, 마지막 오류를 조회한다.
	- `POST /api/system/external-alert-monitor/run`
		- 외부 health URL을 점검한다.
		- foxya DB 접근이 가능하면 `internal_transfers`, `token_deposits`, `external_transfers`, `swaps`, `exchanges`, `payment_deposits` 신규 row를 감지해 텔레그램으로 보낸다.
		- 첫 실행은 현재 max id를 cursor로 저장하고 과거 데이터는 발송하지 않는다.

## 5. 주요 상태값

### 5.1 출금 상태
- `requested`
	- 출금 생성 직후 상태
- `pending_admin`
	- 관리자 최종 승인 대기 상태
- `review_required`
	- 수동 검토가 필요한 상태
- `approved`
	- 요구 승인 수를 충족한 상태
- `broadcasted`
	- 온체인 전송 후 `txHash`가 기록된 상태
- `confirmed`
	- 온체인 확정이 반영된 상태
- `failed`
	- 복구가 완료된 실패 상태
- `rejected`
	- 현재 도메인에는 예약만 되어 있고 별도 reject API는 아직 없다

### 5.2 Sweep 상태
- `planned`
- `broadcasted`
- `confirmed`
- `failed`
- `skipped`

## 7. 현재 제약
- foxya DB signer 정보와 `ENCRYPTION_KEY`가 없으면 sweep bot은 동작하지 않는다.
- source wallet에 TRX gas가 부족하면 sweep bot은 실패로 기록하고 알림만 보낸다.
- foxya `watch-addresses` 응답에는 통화 코드가 없어 같은 주소를 여러 TRON 통화가 공유하면 `DEPOSIT_MONITOR_CURRENCY_IDS` 설정이 필요하다.
- 실제 멀티시그 서명과 관리자 인증/인가 체계는 별도 인증 계층이 추가되어야 완성된다.
