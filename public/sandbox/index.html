    API Capability Guide
    
            read first
          
    
          
    
            
    
              
    Status APIs
    
              
    /health checks process health. /api/system/status explains runtime mode, wallets, monitor collector state, limits, DB, active contract profile, and fixed network presets.
    
            
    
            
    
              
    Monitoring APIs
    
              
    POST /api/system/monitoring/run forces one wallet monitoring collection cycle now. Normal page refreshes only read the latest stored snapshot instead of hitting TRON live on every request.
    
            
    
            
    
              
    On-Chain APIs
    
              
    GET /api/onchain/networks/:network/wallets/:address/balance reads actual TRON TRX and KORI balances for any address on the selected mainnet or testnet. POST /api/onchain/networks/:network/transfers currently sends KORI from the hot wallet only when sandbox send is enabled and the gateway mode is trc20. Multisig cold or treasury wallets are monitor-only in this runtime.
    
            
    
            
    
              
    Deposit API
    
              
    POST /api/deposits/scan simulates detected inbound token movement. If toAddress is a tracked wallet, the amount is credited to the target userId.
    
            
    
            
    
              
    Wallet APIs
    
              
    POST /api/wallets/address-binding binds a TRON address to a user. Then GET /api/wallets/balance and POST /api/wallets/transfer can resolve either userId or walletAddress. This transfer path is internal ledger movement only, not an on-chain TRON send.
    
            
    
            
    
              
    Withdrawal APIs
    
              
    POST /api/withdrawals creates a withdrawal request and locks funds. Then approve, broadcast, and confirm advance the state machine. Only the broadcast step touches the TRON gateway, and that becomes real on-chain signing only when TRON_GATEWAY_MODE=trc20.
    
            
    
            
    
              
    Scheduler API
    
              
    POST /api/scheduler/retry-pending scans stuck withdrawals and tries to reconcile pending broadcast status after the configured timeout window.

    2. Wallet
    
          
    Wallet & Internal Ledger
    
        
    
        
    지갑 바인딩, 내부 잔액, 수동 입금 반영, 내부 이체를 묶어서 검증합니다.
    
      
    

      
    
        
    
          
    
            
    Internal Ledger Balance
    
            GET
          
    
          
    
            Reads the internal ledger only. No blockchain request happens here. Use this after deposit, transfer, or
            withdrawal actions to confirm resulting balances and locked balance.
          
    
          
    
            
            
            
          
    
          
    
        
    

        
    
          
    
            
    Deposit Scan
    
            POST
          
    
          
    
            Simulates a blockchain monitor feeding one detected deposit into the server. If the destination wallet is
            tracked, the amount is credited to the selected userId or the user bound to the beneficiary
            wallet address.
          
    
          
    
            
            
            
            
            
            
            
          
    
          
    
        
    

        
    
          
    
            
    Internal Ledger Transfer
    
            POST
          
    
          
    
            Internal user-to-user ledger transfer. This does not need a wallet address or private key because it does
            not touch TRON. It simply moves balance between two internal accounts identified by userId or
            their bound wallet addresses.
          
    
          
    
            
            
            
            
            
            
            
          
    
          
    
        
    
      
    

      
    
        
    
          
    3. Withdraw
    
          
    Withdrawal & Queue
    
        
    
        
    출금 상태 전이와 scheduler 복구, 승인 대기 큐를 검증합니다.
    
      
    

      
    
        
    
          
    
            
    Withdrawal Control
    
            stateful
          
    
          
    
            This is the full withdrawal state machine. Request locks funds, approve moves it to operator-ready state,
            broadcast hands it to the TRON gateway, and confirm finalizes the ledger. If the gateway mode is
            mock, broadcast is simulated. If it is trc20, the hot wallet private key signs
            the TRON transaction.
          
    
          
    
            
            
            
            
            
            
          
    

          
    
            
            
            
            
            
          
    
          
    
        
    

        
    
          
    
            
    Scheduler
    
            maintenance
          
    
          
    
            Runs the pending reconciliation path. Use this when you want to emulate background recovery for withdrawals
            that stayed too long in pending broadcast state.
